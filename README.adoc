= Maven Central Portal Publisher

image:https://img.shields.io/gradle-plugin-portal/v/io.github.zenhelix.maven-central-publish[Gradle Plugin Portal,link=https://plugins.gradle.org/plugin/io.github.zenhelix.maven-central-publish]

Gradle plugin for publishing to Maven Central via https://central.sonatype.org/publish/publish-portal-api/[Publisher API].

== Installation

[source,kotlin]
----
plugins {
    id("io.github.zenhelix.maven-central-publish") version "{versionPlaceholder}"
}
----

== Overview

Plugin applies `maven-publish` and `signing` plugins automatically, then extends publishing workflow with Maven Central Portal integration:

* Creates checksum files (MD5, SHA1, SHA256, SHA512) for all artifacts
* Packages artifacts, signatures, and checksums into deployment bundles (ZIP)
* Automatically splits large bundles into chunks (configurable max size, default 256 MB)
* Uploads bundles via Publisher API with configurable timeouts
* Polls deployment status until completion
* Automatically drops deployment on failure or timeout
* Retries on HTTP 429 (Too Many Requests) and 5xx errors with exponential backoff
* Auto-registers `javadocJar` and `sourcesJar` tasks (uses Dokka output if available)
* Provides POM metadata DSL with license presets (`apache2()`, `mit()`, `bsd2()`, `bsd3()`) and `scm { fromGithub() }` shortcut

Supports both single-module and multi-module projects with automatic aggregation.
Works with late-configured publications (Kotlin Multiplatform, Android Gradle Plugin) that register publications in `afterEvaluate`.

== Configuration

[source,kotlin]
----
mavenCentralPortal {
    credentials {
        bearer { token = providers.environmentVariable("MAVEN_TOKEN") }
        // OR
        usernamePassword {
            username = providers.environmentVariable("MAVEN_USERNAME")
            password = providers.environmentVariable("MAVEN_PASSWORD")
        }
    }
    publishingType = PublishingMode.AUTOMATIC
    deploymentName = "my-deployment"

    uploader {
        maxStatusChecks = 20
        statusCheckDelay = Duration.ofSeconds(10)
        maxBundleSize = 256.megabytes
        requestTimeout = Duration.ofMinutes(5)
        connectTimeout = Duration.ofSeconds(30)
        maxRetries = 3
        retryBaseDelay = Duration.ofSeconds(2)
    }

    pom {
        name.set("My Library")
        description.set("Library description")
        url.set("https://github.com/user/repo")
        license { apache2() }
        developer {
            id.set("userid")
            name.set("User Name")
            email.set("user@example.com")
        }
        scm { fromGithub("user", "repo") }
    }

    autoConfigureJars = true
}
----

=== Plugin Properties

[cols="1,3"]
|===
|Property |Description

|`baseUrl`
|Maven Central Portal API base URL +
Default: `https://central.sonatype.com`

|`publishingType`
|`PublishingMode.AUTOMATIC` - auto-publish after validation +
`PublishingMode.USER_MANAGED` - manual publish via Portal UI +
Default: `AUTOMATIC`

|`deploymentName`
|Optional deployment identifier in Portal +
Default: `null` (auto-generated)

|`autoConfigureJars`
|Automatically registers `javadocJar` and `sourcesJar` tasks and attaches them to all Maven publications. Uses Dokka output for javadoc if the Dokka plugin is applied. Skipped for Kotlin Multiplatform projects. +
Default: `true`
|===

=== Credentials

Exactly one credential mode must be configured — either `bearer` or `usernamePassword`.
Configuring both will result in a validation error at publish time.

[cols="1,3"]
|===
|Property |Description

|`credentials.bearer.token`
|Bearer token for authentication

|`credentials.usernamePassword.username`
|Maven Central username or token username

|`credentials.usernamePassword.password`
|Maven Central password or token password
|===

Generate tokens at https://central.sonatype.com/account[Maven Central Portal] (User Token section).

=== Uploader Settings

[cols="1,3"]
|===
|Property |Description

|`uploader.maxStatusChecks`
|Max status polling attempts +
Default: `20`

|`uploader.statusCheckDelay`
|Delay between status checks +
Default: `10 seconds`

|`uploader.maxBundleSize`
|Maximum size of a single upload bundle. Bundles exceeding this limit are automatically split into smaller chunks. Use `Int.megabytes` or `Int.gigabytes` extension properties for readable values. +
Default: `256.megabytes`

|`uploader.requestTimeout`
|Timeout for individual HTTP requests to the Maven Central Portal API +
Default: `5 minutes`

|`uploader.connectTimeout`
|Timeout for establishing an HTTP connection +
Default: `30 seconds`

|`uploader.maxRetries`
|Maximum retry attempts for retriable HTTP failures (5xx, connection errors) +
Default: `3`

|`uploader.retryBaseDelay`
|Base delay between retry attempts (exponential backoff: `baseDelay * 2^(attempt-1)`, capped at 5 minutes) +
Default: `2 seconds`
|===

=== POM Metadata

The `pom { }` block provides a convenient DSL for configuring POM metadata required by Maven Central.
When configured, these values are automatically applied to all Maven publications.

[cols="1,3"]
|===
|Property |Description

|`pom.name`
|Project name

|`pom.description`
|Project description

|`pom.url`
|Project URL

|`pom.inceptionYear`
|Project inception year

|`pom.license { }`
|License entry. Supports presets: `apache2()`, `mit()`, `bsd2()`, `bsd3()`. +
Multiple licenses can be added by calling `license { }` multiple times.

|`pom.developer { }`
|Developer entry with `id`, `name`, `email`, `url`. +
Multiple developers can be added by calling `developer { }` multiple times.

|`pom.scm { }`
|SCM connection info. Supports `fromGithub("owner", "repo")` shortcut. +
Or set `connection`, `developerConnection`, `url` manually.
|===

NOTE: If `pom { }` is configured, you don't need to duplicate POM metadata in `publishing { publications { ... pom { } } }`.

== Tasks

Plugin registers tasks per publication and aggregation tasks:

=== Per-Publication Tasks

* `checksum<Publication>Publication` - Generate checksums
* `zipDeployment<Publication>Publication` - Create ZIP bundle
* `publish<Publication>PublicationToMavenCentralPortalRepository` - Upload bundle

=== Aggregation Tasks

* `checksumAllPublications` - Generate checksums for all publications
* `zipDeploymentAllPublications` - Bundle all publications
* `publishAllPublicationsToMavenCentralPortalRepository` - Upload all publications

=== Multi-Module Tasks

For root project with subprojects:

* `zipDeploymentAllModules` - Bundle all modules
* `publishAllModulesToMavenCentralPortalRepository` - Upload all modules

== Lifecycle Task Behavior

Plugin integrates with Gradle's `publish` lifecycle task in two modes:

=== Mode 1: Independent Publishing (default)

Each project that applies the plugin publishes its own bundle independently.
This is the default when the plugin is applied to individual projects (including via convention plugins in `buildSrc`).

[source,bash]
----
./gradlew publish
----

Each project executes its own `publishAllPublicationsToMavenCentralPortalRepository`.

NOTE: When multiple subprojects publish independently, a warning is emitted suggesting atomic aggregation.

=== Mode 2: Atomic Aggregation

When the plugin is applied to the *root project* and subprojects have publications, all modules are aggregated into a single deployment bundle:

[source,bash]
----
./gradlew publish
----

Executes: `publishAllModulesToMavenCentralPortalRepository` (root only).
Subprojects do not publish independently — one bundle, one deployment, atomic release.

To enable this mode, apply the plugin to the root project and configure credentials there.

== Usage

=== Single Module

[source,kotlin]
----
mavenCentralPortal {
    credentials {
        bearer { token = providers.environmentVariable("MAVEN_TOKEN") }
    }
    publishingType = PublishingMode.AUTOMATIC

    pom {
        name.set("My Library")
        description.set("Library description")
        url.set("https://github.com/user/repo")
        license { apache2() }
        developer {
            id.set("userid")
            name.set("User Name")
            email.set("user@example.com")
        }
        scm { fromGithub("user", "repo") }
    }
}

publishing {
    publications {
        create<MavenPublication>("mavenJava") {
            from(components["java"])
        }
    }
}

signing {
    val signingKey: String? by project
    val signingPassword: String? by project
    useInMemoryPgpKeys(signingKey, signingPassword)
    sign(publishing.publications)
}
----

[source,bash]
----
./gradlew publishAllPublicationsToMavenCentralPortalRepository
----

=== Multi-Module

Apply plugin to root and subprojects.
Root task aggregates all modules:

[source,bash]
----
./gradlew publishAllModulesToMavenCentralPortalRepository
----

=== Convention Plugin (buildSrc)

Common pattern using a convention plugin in `buildSrc`:

.buildSrc/src/main/kotlin/my-publishing.gradle.kts
[source,kotlin]
----
plugins {
    id("io.github.zenhelix.maven-central-publish")
}

mavenCentralPortal {
    credentials {
        bearer { token = providers.environmentVariable("MAVEN_TOKEN") }
    }

    pom {
        url.set("https://github.com/user/repo")
        license { apache2() }
        developer {
            id.set("userid")
            name.set("User Name")
            email.set("user@example.com")
        }
        scm { fromGithub("user", "repo") }
    }
}

signing {
    val signingKey: String? by project
    val signingPassword: String? by project
    useInMemoryPgpKeys(signingKey, signingPassword)
    sign(publishing.publications)
}
----

.subproject/build.gradle.kts
[source,kotlin]
----
plugins {
    `java-library`
    id("my-publishing")
}

publishing {
    publications {
        create<MavenPublication>("mavenJava") {
            from(components["java"])
        }
    }
}
----

Each subproject publishes independently. For atomic multi-module publishing, also apply the plugin to the root project.

== Requirements

* Gradle 8.0+
* Java 17+
* Valid Maven Central account with a https://central.sonatype.com/account[bearer token] or username/password credentials
* POM metadata: name, description, url, licenses, developers, scm (can be configured via the `pom { }` DSL)
* Signed artifacts

== Development

.Click to expand
[%collapsible]
=====
*CI/CD Pipeline*

The project uses GitHub Actions with reusable workflows from https://github.com/zenhelix/ci-workflows[zenhelix/ci-workflows] v2.

[cols="1,2"]
|===
|Workflow |Purpose

|`build-on-branch.yml`
|Runs `check` on pull requests and auto-labels PRs based on changed file paths

|`build-on-main.yml`
|Runs `check` and creates version tag on push to `main`

|`release-on-tag.yml`
|Creates GitHub Release and publishes to Gradle Plugin Portal and Maven Central
|===

PR labeling rules are configured in `.github/labeler.yml`. +
Changelog categories are configured in `.github/changelog-config.json`.

*Release Flow*

[source]
----
PR merge → check → create tag → [tag push] → release → publish
----

*Versioning via Commit Messages*

Version bumps are controlled by tokens in commit messages:

[cols="1,1,2"]
|===
|Token |Bump |Example

|`#major`
|X.0.0
|1.0.0 → 2.0.0 (Breaking changes)

|`#minor`
|0.X.0
|1.0.0 → 1.1.0 (New features)

|`#patch` or none
|0.0.X
|1.0.0 → 1.0.1 (Fixes, default)

|`#none`
|—
|Skip tag creation
|===

Example commit message:

[source]
----
feat: Add new authentication method #minor
----

*Regular Release*

. Create PR to `main`
. Merge with version token in commit message (or omit for patch)
. Pipeline automatically: check → tag → release → publish

*Creating Major Version (1.x → 2.x)*

When releasing breaking changes:

[source,bash]
----
# Create support branch for current version
git checkout main && git pull
git checkout -b 1.x
git push origin 1.x

# Return to main and merge PR with #major token
git checkout main
# Merge PR with commit message containing #major
# Auto-creates tag 2.0.0
----

*Backporting to Old Version*

[source,bash]
----
# Cherry-pick fix to support branch
git checkout 1.x && git pull
git cherry-pick <commit-hash>
git checkout -b fix/security-patch-1x
git push origin fix/security-patch-1x

# Create PR: fix/security-patch-1x → 1.x
# After merge: auto-creates tag 1.x.y → release → publish
----

*Rollback (before publish)*

[source,bash]
----
git push --delete origin 1.0.1
git tag -d 1.0.1
----

NOTE: Published versions cannot be deleted from Gradle Plugin Portal.
